# Azure Management VM (Jumpbox) - Standalone Project

Terraform project that deploys a fully network-isolated Linux VM for ad-hoc scripts, troubleshooting, and verification tasks. This project is **independent** from the network project (`azure-vnet-terraform`) - it attaches into an existing subnet by ID, but never creates, modifies, or destroys the VNet, subnets, NSGs, or any other networking resource.

## Why this design

This VM has no public IP and no inbound network exposure of any kind. It was deliberately built this way instead of using a traditional SSH bastion or the managed Azure Bastion service, trading interactive convenience for a minimal attack surface and zero ongoing service cost.

Access works through two Azure control-plane mechanisms instead of network connectivity:

| Mechanism | Use case | Requires network path to the VM? |
|---|---|---|
| `az vm run-command` | One-off commands, scripts, automation | No - goes through Azure Resource Manager |
| Azure Serial Console | Interactive shell session, boot diagnostics | No - goes through the Azure control plane |

Neither mechanism touches the VNet's data plane. The VM's NSG (inherited from the subnet it's deployed into) can keep denying all inbound traffic with zero exceptions.

## Architecture

```
azure-vnet-terraform/  (separate project, deployed first)
└── snet-app-az1  (existing subnet, referenced by ID)
        │
        │ var.app_subnet_id (passed explicitly, no remote state read)
        ▼
azure-jumpbox/  (this project)
├── nic-mgmt-jumpbox    (private IP only, no public IP)
└── vm-mgmt-jumpbox     (Ubuntu 22.04 LTS)
```

## What this deploys

| Resource | Notes |
|---|---|
| Network Interface | `nic-mgmt-jumpbox`, private IP only, attached to the existing subnet passed via `app_subnet_id` |
| Linux Virtual Machine | `vm-mgmt-jumpbox`, Ubuntu 22.04 LTS, no public IP |
| SSH key pair | Generated by Terraform (`tls_private_key`), satisfies Azure's API requirement for VM creation, never written to disk, never used for actual access |

This project does **not** create a resource group, VNet, subnet, or NSG - those belong to the network project and are referenced as a data source (`data.azurerm_resource_group.existing`).

## Prerequisites

- [Terraform](https://developer.hashicorp.com/terraform/downloads) >= 1.5.0
- [Azure CLI](https://learn.microsoft.com/en-us/cli/azure/install-azure-cli), logged in via `az login`
- The network project (`azure-vnet-terraform`) already deployed - you need its `resource_group_name` and `app_subnet_ids` outputs

## Getting the required inputs from the network project

From the `azure-vnet-terraform` directory:

```bash
terraform output -raw resource_group_name
terraform output -json app_subnet_ids
```

Copy the resource group name and the first subnet ID (index 0, AZ1) from those outputs.

## Configuration

Create a `terraform.tfvars` file in this directory (already excluded by `.gitignore`):

```hcl
resource_group_name = "rg-ha-vnet"
app_subnet_id        = "/subscriptions/<sub-id>/resourceGroups/rg-ha-vnet/providers/Microsoft.Network/virtualNetworks/vnet-ha/subnets/snet-app-az1"
```

| Variable | Default | Description |
|---|---|---|
| `resource_group_name` | `rg-ha-vnet` | Name of the EXISTING resource group from the network project |
| `location` | `eastus` | Must match the existing VNet's region |
| `app_subnet_id` | *(required, no default)* | Full resource ID of the existing App subnet to deploy into |
| `mgmt_vm_size` | `Standard_B1s` | VM size |
| `mgmt_admin_username` | `azureuser` | Local admin username on the VM |
| `mgmt_zone` | `1` | Availability zone - should match the AZ of `app_subnet_id` |

## Deploy

```bash
terraform init
terraform plan
terraform apply
```

This creates only the NIC and VM (plus the disposable SSH key) - it never touches the network project's resources.

## Accessing the VM

### 1. One-off commands via the Azure VM Agent

No setup required - works immediately after deployment.

```bash
az vm run-command invoke \
  --resource-group rg-ha-vnet \
  --name vm-mgmt-jumpbox \
  --command-id RunShellScript \
  --scripts "hostname && uptime"
```

Use this for scripts, configuration tasks, and verification commands. Each invocation is a single round trip - there's no persistent interactive session through this path.

### 2. Interactive shell via Azure Serial Console

Serial Console gives you a real interactive terminal, but it logs in like a physical console - it needs a local username and password on the VM (SSH keys don't apply here, since this isn't SSH).

**One-time setup - set a password for the admin user:**

```bash
az vm run-command invoke \
  --resource-group rg-ha-vnet \
  --name vm-mgmt-jumpbox \
  --command-id RunShellScript \
  --scripts 'echo "azureuser:YOUR_PASSWORD_HERE" | chpasswd'
```

> Use single quotes around the whole `--scripts` value if your password contains `!` or other shell special characters - otherwise your local shell may try to interpret them (e.g. bash history expansion) before the command ever reaches Azure.

**Connect to Serial Console:**

Via Azure CLI:
```bash
az extension add --name serial-console   # one-time, if not already installed
az serial-console connect \
  --resource-group rg-ha-vnet \
  --name vm-mgmt-jumpbox
```

Or via the Azure Portal: VM resource -> Connect -> Serial console.

At the `login:` prompt, enter `azureuser` and the password you set above.

To exit the Serial Console session: `Ctrl+]` then `q`.

> Serial Console authenticates against the VM's local OS login (`/etc/shadow`), which is independent of SSH. Setting `disable_password_authentication` on the VM resource (which this project does, for SSH) does not affect Serial Console login - it only affects SSH password authentication. The VM has no SSH path reachable from outside the VNet regardless.

### Why not plain SSH?

The VM has no public IP, and the subnet's NSG denies all inbound traffic. There is no network path for SSH to reach this VM at all - the two mechanisms above are the only supported access methods by design.

## Sharing this project with others

If someone else needs to deploy or manage this jumpbox:

1. They need the network project's outputs (`resource_group_name`, `app_subnet_ids[0]`) - get these from whoever manages `azure-vnet-terraform`, or from a shared output if using remote state.
2. They create their own `terraform.tfvars` with those values - never commit this file (already covered by `.gitignore`).
3. After `terraform apply`, they set their own password via `chpasswd` (see above) rather than reusing someone else's - each operator should set their own credential.
4. Anyone needing Serial Console access needs the `Virtual Machine Contributor` (or higher) Azure RBAC role on this VM, since Serial Console access itself is gated by Azure RBAC in addition to the OS-level login.

## Destroying

```bash
terraform destroy
```

This only removes the NIC and VM created by this project - the VNet, subnets, NSGs, and all other network resources remain untouched.

## Security notes

- No public IP, no open inbound ports - the smallest possible network attack surface for a VM that still needs management access.
- The SSH key pair generated by this project is never used and never written to disk - it exists only to satisfy Azure's VM creation API.
- The Serial Console password is set out-of-band via `run-command`, not stored in Terraform state or any `.tf` file - rotate it periodically using the same `chpasswd` command.
- Outbound internet access (e.g. for `apt-get update`) flows through the network project's shared NAT Gateway, associated with this subnet.
